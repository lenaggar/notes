- `ip addr` or `ip a` => view ip addresses for different interfaces

	- `eth0` or `enp1s0` => ethernet interface
	- `wlan0` or `wlp1s0` => wifi interface
	- funny thing about loopback is that we can actually `ssh` into our own ssh server by running `ssh mahmoud@127.0.0.1` and in this case we have an ssh session on the same machine without actually leaving the machine :D
- `ip route` => view routing table, the default route (`default via 192.168.1.1`) is our gateway (home router) where traffic goes to reach the internet
  in the cloud it is a bit more complicated than on a home network, in the cloud the route goes through different multiple routers before it leaves to go to the internet
- `hostname` => display the hostname of our server (the hostname is the name of the server on the network)
- `hostnamectl` => display detailed hostname and OS architecture info
- `cat /etc/hosts` => display a lookup table for ip addresses and their names on the server, this is where the infamous `localhost` is set
	- so we can `ssh` to our own server by running `ssh mahmoud@localhost` same as above
- `ping google.com` => pings to check network connectivity, will keep pinging indefinitely
- `ping -c 4 google.com` => pings to check network connectivity, pings only 4 times then stops
- `curl https://example.com` => displays the content HTML
- `curl -I https://example.com` => show just the headers, useful for checking if a site is up
- `curl -O https://example.com/file.txt` => downloads a file form the internet with the same remote name (-O uses remote filename)
- `curl -o my-file.md https://example.com/file.txt` => downloads a file form the internet with a custom name
- `wget https://example.com/file.txt` => downloads a file form the internet (same as `curl -O`)
- `ss -tuln` => view network connections
	- `-t` => TCP

	- `-l` => Listening (server) sockets - processes that are listening to network ports

## SSH

So what is `ssh` reall?

`ssh` stands for (Secure Shell), where the `ssh client` creates an `encrypted` connection between your computer and the `ssh server` on the remote server. Everything that is typed in and outputted back is encrypted.
It was designed to be a replacement for `telnet` which was not encrypted and hence was insecure, anyone in the middle can see the traffic.

So, the key components here are:
- `SSH Client` => on the computer (that's accessed through the `ssh` command)
- `SSH Server` => on the remote server (that's the `sshd.service` running on the server)
- `Encryption` => Traffic is unreadable to anyone in between

### How does SSH work
1. user runs `ssh user@server`
2. client connects to server on port `22`
3. client and server negotiate encryption (key exchange)
4. user authenticates (password or key)
	- Password Authentication:
		- simple
		- less secure
		- password is types every time
		- vulnerable to brute force attacks
	- Key authentication:
		- more secure
		- no password needed once setup in (usually public key in `~/.ssh/`)
		- based on cryptography key pairs
		- required by many cloud providers
5. encryption tunnel is established and a shell session is started
6. command run on the server

### Generating an SSH key
1. On the Client:
	- `ssh-keygen -t ed25519 -C "your_email@example.com"` => generate a new ssh key using `ed25519` algorithm and `"your_email@example.com"` email. This outputs two files:
		- `~/.ssh/id_ed25519`: (which is the secret key that should never be shared with anyone)
		- `~/.ssh/id_ed25519.pub`: (which is a public key that is safe to share with services or servers to present them with your identity)
	- `ssh-add ~/.ssh/id_ed25519` => add identity to the `ssh client` on your machine
	- `ssh-copy-id user@server` => add keys to the server as trusted keys to `~/.ssh/authorized_keys` file
	- `ssh -v user@server` => `-v` for verbose when establishing the connection
	- `sudo vi ~/.ssh/config` then modify:```
			Host Naggar
				HostName 192.168.100.71
				User mahmoud
				IdentityFile ~/.ssh/id_ed25519```
		- this makes connecting to the server more convenient by just typing:
		- `ssh Naggar` and the ssh client will do the rest, it will know the ip address of the server, the username, and the authentication key to use
	- `chmod 600 ~/.ssh/config` => set proper permissions for the ssh key config file on the client

2. On the Server:
	- `sudo vi /etc/ssh/sshd_config` then modify:
		- `PasswordAuthentication no`
		- `PubkeyAuthentication yes`
		- this is more secure, now the server won't allow any password authentications, only public key authentications
## scp
- `scp file.txt user@server:/home/user/` => copy local file to remote server
- `scp user@server:/var/log/syslog ./` => copy remote file to local directory
- `scp -r directory/ user@server:/home/user/` => copy directory recursively to remote server

## SSH best practices
- use keys, not passwords - disable password auth whenever possible
- use a passphrase - protect the key if someone got the file
- use a config file - simpler commands, fewer mistakes
- keep private keys private - never share, they are only intended for the ssh client
