Bash is a full blown programming language. It has variables, conditionals, loops and functions

Scripts are a way of making repeatable steps easier to call repeat

In a shell script the first line is the `Shebang`
it tells the system before running the script, which interpreter to use

- `#!/bin/bash` => use bash
	- use `#!/bin/bash` not `#!/usr/bin/env bash` because it is a security risk!
	  the latter searches for `bash` in `PATH`. so, an attacker can put malicious `bash` in `PATH`. with the first one, we know exactly what runs

Do not use shell extension `hello` > `hello.sh`. the Shebang already tells the system what interpreter to use. Real unix tools don't have extensions `ls`, `grep`, `vim`

In order to make a shell script executable:
- run `chmod +x hello` => now it becomes executable with `./hello`

## PATH
`PATH` is a special variable that contains concatenated (by `:`) directories for the shell to use in sequence to look for executables
So, when we run `ls`, the shell goes through each directory inside the `PATH` variable to search for an executable called `ls`, the first one it finds will be executed

`echo $PATH` => shows the variable

It should be noted that putting `./` in the `PATH` variable is potential security risk and an anti-pattern, executables from current directory should be invoked/run by pre-pending `./` to the executable name (`./hello`)

- `bash -x hello` => run in debug mode
	- shows variable expansion, command execution
	- before asking for help why a script is failing, run in debug mode `bash -x` often times the problem will be spotted easily

- `shellcheck` => _**IMPORTANT**_, checks shell script for bugs, security issues
	- will even include wiki or docs about the rules that are broken

## Example

1. Display system info:
```bash
#!/bin/bash

# Description: Display system information

echo "=== System Information ==="
echo "Hostname: $(busybox hostname)"
echo "Date: $(date)"
echo "Kernel: $(uname -r)"
```

2. Dotfiles setup
	- first create `~/.config`
	- move all the dotfiles to share inside
	- create `~/dotfiles/setup` as shown below, and now this becomes a shareable dotfiles directory that can be hosted publicly on github
	- steps to reuse on another green field server:
		1. clone the repo to `~/dotfiles`
		2. run `~/dotfiles/setup`
```bash
#!/bin/bash

# Description: Link dotfiles to home directory

mkdir -p ~/.config

ln -sf ~/dotfiles/bashrc ~/.bashrc
ln -sf ~/dotfiles/vimrc ~/.vimrc
ln -sf ~/dotfiles/starship.toml ~/.config/starship.toml

echo "Dotfiles linked!"
```
