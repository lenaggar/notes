initialization
```bash
#!/bin/bash
name="DevOps"  # correct
name = "DevOps"  # wrong
```

evaluation `${variable}`
```bash
#!/bin/bash
echo $name  # correct, but not recommended
echo ${name}  # correct, do this
```

evaluated inside strings (template literals)
```bash
#!/bin/bash
name="World"
echo "Hello, ${name}!"  # corect
echo 'Hello, ${name}!'  # wrong, not evaluated, nothing expands, literal text

message="Hello      World"
echo ${message}
# Hello World (spaces copllapsed)
echo "${message}"
# Hello      World (spaces preserved)

files="*.txt"
echo ${files}
# names.txt users.txt (expanded!)
echo "${files}"
# *.txt (literal)
```

Always put variables in double quotes `"` (Avoid unpredictable evaluations)
```bash
#!/bin/bash
touch "my important file.txt"

file="my important file.txt"
rm ${file}  # wrong - tries to do (rm my, rm file, rm file.txt) - worse, it can delete the wrong files. `shellcheck` can be useful to catch these before hand
rm "${file}"  # correct - does exactly what we expect
```

Command substitution `$(command)`
```bash
#!/bin/bash
today=$(date +%Y-%m-%d)
echo "Today is ${today}"  # Today is 2026-04-12

branch=$(git branch --show-current)
echo "Active git branch: $(branch)"  # Active git branch: main

dir=$(pwd)
echo "Directory: $(dir)"  # Directory: /home/mahmoud/lab/bash/02.variables
```

Special characters (`${0}`, `${1}`, `${@}`,`${#}`)
```bash
#!/bin/bash
echo "Script name: ${0}"
echo "First argument: ${1}"
echo "All arguments: ${@}"
echo "Number of arguments: ${#}"
```

| Variable        | Meaning                                               |
| --------------- | ----------------------------------------------------- |
| `$0`            | script name                                           |
| `$1`, `$2`, ... | positional arguments                                  |
| `$@`            | all arguments as words space-separated - **Use this** |
| `$*`            | all arguments as single string - **Avoid this**       |
| `$#`            | number of arguments                                   |
| `$?`            | holds the exit code of the last command               |
| `$$`            | current shell's pid                                   |
| `$USER`         | current username                                      |
| `$HOME`         | home directory of current user                        |
| `$PWD`          | current directory                                     |
 
Special characters (`${@}` vs `${*}`)
```bash
# if called with ./script "hello world" "foo bar"
# these are two arguments ^

echo "${@}"  # returns two arguments: "hello world" "foo bar"
echo "${*}"  # returns one arguments: "hello world foo bar"
```

Special characters (`${?}`)
```bash
#!/bin/bash
ls /etc/passwd
echo "Exit code: ${?}"  # 0 (success)

ls /nonexistent
echo "Exit code: ${?}"  # 2 (failure)
```
In bash, a `0` exit code means that the command exited with `0` errors, that's why it means success, any non-`0` number means failure
This is how chaining commands `&&` and `||` work.

Environment variables
```bash
#!/bin/bash
local_var="local"          # Local variable - only in current shell
export ENV_VAR="exported"  # Environment variable - inherited by child processes
```
Convention is to use capitalized letters for variables exported to child processes.
**Security Note**: Environment variables are visible via `/proc/<pid>/environ` and `ps e`. For secrets, use files with _restricted permissions_ instead.
