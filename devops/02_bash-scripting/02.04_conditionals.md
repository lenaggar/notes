```bash
ls /etc/passwd
echo "Exit code: ${?}"  # 0 - success

ls /nonexistent
echo "Exit code: ${?}"  # 2 - failure
```

> _"Everything in Unix returns an exit code.
> 	`0` = success
> 	`Non-zero` = failure (1-255)
> This fundamental to how UNIX work and is the foundation of all conditionals"_

This is how `&&` and `||` work. They check exit codes.

Manually setting exit code
```bash
#!/bin/bash

if [[ ! -f "$1" ]]; then
    echo "File not found: $1"
    exit 1
fi

echo "Processing $1..."
exit 0
```

## Logical operators
### AND (`&&`)
Run the second command **ONLY** if the first succeeds
```bash
mkdir mydir && cd mydir
git pull && echo "Pull successful"
```

### OR (`||`)
Run the second command only if the first fails
```bash
cd mydir || mkdir mydir
command -v docker &>/dev/null || sudo apt install docker
```

### Chaining
```bash
cd project || mkdir project && cd project
./run-tests && ./deploy || echo "Tests failed!"
```

## Conditional brackets `[]`, `[[]]`
> _"Single brackets `[]` are a security vulnerability.
> Double brackets `[[]]`. Always. No exceptions.”_

```bash
# WRONG - single brackets have many gotchas
if [ $name == "DevOps" ]; then

# CORRECT - double brackets are safer
if [[ "$name" == "DevOps" ]]; then
```

Single brackets `[]` is a shell builtin (synonym `test`). They don’t prevent word splitting.
- `help [` => This is a synonym for the `test` shell builtin, but the last argument must be a literal `]`, to match the opening `[`
```bash
name="hello world"

[ $name == "hello world" ]    # ERROR: too many arguments
[[ $name == "hello world" ]]  # No error
```
> _“I’ve seen production outages caused by single brackets and unquoted variables.
> This isn’t academic - it’s job security.”_

Bracket syntax rules:
```bash
# Spaces are required around brackets and operators
[[ "$name" == "DevOps" ]]  # CORRECT
[["$name"=="DevOps"]]      # WRONG - no spaces around ==
```

## String comparisons
```bash
#!/bin/bash

str1="hello"
str2="world"

[[ "$str1" == "$str2" ]] && echo "Equal"
[[ "$str1" != "$str2" ]] && echo "Not equal"

# Empty/not empty
[[ -z "$name" ]] && echo "Empty"
[[ -n "$name" ]] && echo "Not empty"
```

| Operation | Meaning                        |
| --------- | ------------------------------ |
| `==`      | is equal                       |
| `!=`      | in not equal                   |
| `-z`      | is empty (zero length)         |
| `-n`      | is not empty (length not zero) |

## Numeric comparisons
```bash
#!/bin/bash

count=5

[[ $count -eq 5 ]] && echo "Equal to 5"
[[ $count -gt 3 ]] && echo "Greater than 3"
[[ $count -lt 10 ]] && echo "Less than 10"
```

| Operation | Meaning                  |
| --------- | ------------------------ |
| `-eq`     | is equal                 |
| `-ne`     | in not equal             |
| `-gt`     | is greater than          |
| `-lt`     | is less than             |
| `-ge`     | is greater than or equal |
| `-le`     | is less than or equal    |
> _“Use `-eq`, `-lt`, `-gt` for numbers.
> Use `==`, `!=` for strings.
> Mix them up and you get weird bugs.”_

### Arithmetic `(( ))` for Numbers
```bash
#!/bin/bash

count=5

if (( count > 3 )); then
    echo "Count is greater than 3"
fi
```
Inside `(( ))`: use `<`, `>`, `==`, no $ needed for variables.

## File tests
```bash
#!/bin/bash

[[ -f /etc/passwd ]] && echo "File exists"
[[ -d /home ]] && echo "Directory exists"
[[ -x /bin/bash ]] && echo "Executable"
```

| Operation | Meaning                              |
| --------- | ------------------------------------ |
| `-f`      | does file exist, and<br>is type file |
| `-d`      | does dir exist, and<br>is type dir   |
| `-e`      | does file/dir exist                  |
| `-r`      | is readable                          |
| `-w`      | is writable                          |
| `-x`      | is executable                        |
| `-s`      | does file exist, and<br>is not empty |

## If/Else statements
Simple `if`, `elif`, `else`
```bash
#!/bin/bash

if [[ -f "$1" ]]; then
	echo "It's a file"
elif [[ -d "$1" ]]; then
    echo "It's a directory"
else
    echo "Not found: $1"
fi
```

Combining conditions
```bash
#!/bin/bash

# AND - both must be true
if [[ -f "$1" && -r "$1" ]]; then
    echo "File exists and is readable"
fi

# OR - either can be true
if [[ -z "$1" || "$1" == "--help" ]]; then
    echo "Usage: $0 <filename>"
fi

# NOT - invert the condition
if [[ ! -f "$1" ]]; then
    echo "Not a file: $1"
fi
```

## Checking command existence
```bash
if command -v docker &>/dev/null; then
	echo "Docker is installed"
else
	echo "Docker is not installed"
fi
```

Why use `command -v` instead of `which`:
- `cammnd-v` => bash builtin (faster)
- `which` => external, may not be installed
- `command -v` => POSIX compliant

```bash
#!/bin/bash

for cmd in git docker kubectl; do
	if ! command -v "${cmd}" &>/dev/null; then
		echo "Error: ${cmd} is not installed"
		exit 1
	fi
done
```
This pattern ^ shows up in every production script. Check your dependencies upfront before the script does real work.

## Case statements
```bash
#!/bin/bash

case $1 in
	start)
		echo "Starting service..."
		;;
	stop)
		echo "Stopping service..."
		;;
	*)
		echo "Usage: $0 {start|stop}"
		exit 1
		;;
esac
```
notes on case statements:
- each patterns ends with `)`
- each block ends with `;;`
- `*` is the default case (matching anything else)
- `esac` ends the case statement
