```bash
docker run hello-world
```

What this command does:

3. Created a container from that image

## Docker commands
- `docker search nginx` => search the docker registry for `nginx` image
- `docker pull nginx` => pulls image from docker registry with `latest` tag
  `docker pull nginx:1.28`, `docker pull nginx:1.28-alpine` => pills tagged images
	- it is important to note that in production environments we should use tagged images, `latest` changes over time
- `docker images` => list images locally
- `docker run nginx` => runs `nginx` in the foreground
  `docker run -d nginx` => runs `nginx` in the background (`-d`: detach mode) returns the container id and gives us back the terminal
  `docker run -d --name webserver nignx` => names the container for easier management instead of relying on docker's random container name generation
- `docker ps` => shows all the _running_ containers on the system
  `docker ps -a` => shows all containers in any state (`running`, `exited`, and `created`)

### Interactive containers
Some containers are meant to be interactive, to interact with them we need a shell inside it through which we can interact with it
- `docker run -it ubuntu bash`  => `-it` gives us the interactive shell
	- `-i` => interactive (keep `stdin` open)
	- `-t` => allocate a pseudo-terminal
	When we exit the container stops (the shell was the main process in this case because we specified it with `bash`)
	If we hadn't specified a 
```bash
# host's shell
$ docker run -it ubuntu bash

# ubuntu container's shell
root@8a441916aa62:/# ls
bin  boot  dev  etc  home  lib  ...

root@8a441916aa62:/# cat /etc/os-release
PRETTY_NAME="Ubuntu 24.04.4 LTS"  ...

# exiting uvbuntu's container's shell
root@8a441916aa62:/# exit
exit

# back to host's shell
```

- `docker run -it --rm ubuntu bash` 
	- `--rm` => removes the container when it exits, used for quick tests in throwaway containers

### Port publishing
Containers have their own network, to access a containerized service  from host:
- `docker run -d -p 8080:80 nginx`
	- format `-p HOST_PORT:CONTAINER_PORT`
	- now port `80` inside the container is accessible for us through `localhost:8080`
	  we can `curl localhost:8080` to verify
- `docker run -d -p 8080:80 -p 8443:443 nginx` => publish multiple ports
- `docker run -d -p 80 nginx` => gets random `HOST_PORT`

### Environment variables
Containers can be configured through environment variables
```bash
docker run -d \
  --name mydb \
  -e POSTGRES_PASSWORD=supersecretpassword \
  -e POSTGRES_USER=myuser \
  -e POSTGRES_DB=myapp \
  postgres
```
These are passed to the postgres container to configure the DB.

1. `docker ps` => see that the container is indeed running.
2. `docker logs mydb` => see the container's logs, and at the end there's this line `database system is ready to accept connections` which means that postgress is actually running and waiting for instructions.
3. `docker exec mydb env` => prints all the environment variables available for postgres and we can see the ones we passed are available

Instead of passing these environment variables inline as we did above, we can put them in a `db.env` file and pass that to the container like so
```bash
echo "POSTGRES_PASSWORD=supersecretpassword" > db.env
echo "POSTGRES_USER=myuser" >> db.env
echo "POSTGRES_DB=myapp" >> db.env

docker run -d --env-file db.env postgres  # better and more secure
```
>_"Never put secrets in command history. Use --env-file or secrets management instead"_

Refer here to [the 12 factor app principles' configuration part](https://12factor.net/config)

### Viewing logs
Running containers have their own logs, and we can check these logs like so:
- `docker logs webserver` => view container logs
- `docker logs -f webserver` => follow logs
- `docker logs --tail 100 webserver` => last 100 lines
- `docker logs -t webserver` => logs with timestamps

### Cleaning up
- `docker stop wbserver` => sends `SIGTERM`, waits, then `SIGKILL`. container is still on disk
- `docker rm webserver` => remove container (must be stopped first of use `-f`)
- `docker rm -f webserver` => stops the container and removes it
- `docker container prune` => remove all stopped containers
- `docker rmi nginx:1.28` => remove an image
- `docker image prune`=> remove all unused images
- `docker system prune -a` => remove everything (stopped containers, unused images, unused networks)
