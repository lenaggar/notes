## What is a Dockerfile
A Dockerfile is a text file containing instructions to build a container image (recipe)
```dockerfile
FROM ubuntu:24.04                              # start with ubuntu
RUN apt-get update && apt-get install -y curl  # install packages
COPY greet /app/                               # copy files in
CMD ["/app/greet/"]                            # default command
```

Important to know that each instruction creates a layer in the image.

---
## Exercise: Build your first docker image
Create a new directory
```bash
mkdir myapp && cd myapp
```

Create a simple `greet` script
```bash:greet
#!/bin/bash
echo "Hello, from my container!"
```

Make it executable
```bash
chmod +x ./greet
```

Create a new `Dockerfile`
```dockerfile
FROM ubuntu:24.04
WORKDIR /app
COPY greet .
CMD ["./greet"]
```

Build the image
```bash
docker build -t myapp:1.0.0 .
```
- `-t myapp:1.0.0` => tag the image with name and version
- `.` => use the current directory as build context

Now run it!
```bash
docker run myapp:1.0.0  # Hello, from my container!
```
>_"Dockerfile + source code = reproducible, portable application."_

## Core Dockerfile instructions
### FROM - Base image

Every Dockerfile starts with `FROM`. It is always the starting point
```dockerfile
FROM ubuntu:24.04          # full-featured linux
FROM debian:bookworm-slim  # smalled debian
FROM alpine:3.20           # minimal (~5MBs)
FROM scratch               # empty, for static libraries
```

Always use a specific tag, not `latest`

### RUN - Execute commands

Run commands during the build process
```dockerfile
RUN apt update && apt install -y curl                   # install packages
RUN mkdir -p /app/data                                  # create directories
RUN curl -p /app/file.txt https://example.com/file.txt  # download files
```

Each `RUN` creates a new layer. It's more efficient to combine related `RUN`s
```dockerfile
# Bad: 3 layers
RUN apt update
RUN apt install -y curl
RUN apt clean

# Good: 1 layer
RUN apt update && \
	apt install -y curl && \
	apt clean
```

#### Bundling dependencies
Containers must include everything the application needs.

For example, this will fail because the we have a missing dependency `ca-certificates`
```bash
# install `curl` without CA certificates - try HTTPS
docker run --rm ubuntu:24.04 bash -c "
	apt update &&
	apt install -y --no-install-recommends curl &&
	curl https://example.com
"
```
We get this error: `curl: (77) error setting certificate file: /etc/ssl/certs/ca-certificates.crt`

Fixed version
```bash
# install `curl` without CA certificates - try HTTPS
docker run --rm ubuntu:24.04 bash -c "
	apt update &&
	apt install -y --no-install-recommends curl ca-certificates &&
	curl https://example.com
"
```
Now this succeeds because the container has all its needed dependencies bundled.

**Lesson**: Containers should always bundle all dependencies explicitly, Nothing is assumed to exist.

### COPY - Copy files

Copy files from the build context into the image
```dockerfile
COPY start /app/                        # copy single file
COPY scripts/ /app/scripts/             # copy directory contents
COPY config.json data.json /app/        # copy multiple files
COPY config.prod.json /app/config.json  # copy with different names
```

### WORKDIR - Set working directory

Sets the working directory for subsequent instructions
```dockerfile
WORKDIR /app

# Now every instuction that follows happen in /app/
COPY ..
RUN chmod +x start
CMD ["./start"]
```
Better than `RUN cd /app` because it persists
### ENV - Environment variables

Set environment variables available at runtime
```dockerfile
ENV APP_ENV=production
ENV PORT=8080
ENV LOG_LEVEL=info
```

### EXPOSE - Document ports

Documents which ports the application is using. Doesn't actually publish them
```dockerfile
EXPOSE 8080
EXPOSE 443
```

This is **ONLY** for documentation. We still need to use `-p` when running the container
```bash
docker run -p 8080:8080 myapp
```

### CMD - Default command

The command to run when a container starts
```dockerfile
CMD ["./start"]  # exec form (preferred)
CMD ./start      # shell form
```

This can however be overridden at runtime
```bash
docker run myapp ./other-script
```

### ENTRYPOINT - Fixed command

Like `CMD` but harder to override. This is good for containers that should always run a specific executable.

#### Example: A backup tool

Create a backup script `backup`
```bash
#!/bin/bash

case "${1}" in
	create) echo "Creating backup..." ;;
	restore) echo "Restoring from backup..." ;;
	list) echo "listing backups..." ;;
	*) echo "Usage: backup {create|restore|list}" ;;
esac
```

Create `Dockerfile` for `backup`
```dockerfile
FROM ubuntu:24.04
WORKDIR /app
COPY backup .
RUN chmod +x backup
ENTRYPOINT ["./backup"]
CMD ["list"]
```

Build the image
```bash
docker build -t backup:1.0.0 .
```

Run the container
```bash
docker run backup:1.0.0           # Runs: ./backup list
docker run backup:1.0.0  create   # Runs: ./backup create
docker run backup:1.0.0  restore  # Runs: ./backup restore
```

So, now the container itself is like the `backup` command.
Arguments that we pass become arguments to the `ENTRYPOINT`.

If we want to get a shell, we must override the `ENTRYPOINT`, like so
```bash
docker run backup:1.0.0 bash                   # runs: ./backup bash (incorrect)
docker run --entrypoint bash backup:1.0.0      # runs: bash (shell access)
docker run -it --entrypoint bash backup:1.0.0  # interactive shell
```

>_"**Rule of thumb**:
>	Use `CMD` for most applications,
>	Use `ENTRYPOINT` when the container is the command."_

## Layers and Caching

Each instruction creates a layer. Docker caches layers to speed up builds.

### How caching works

```dockerfile
FROM ubuntu:24.04           # Layer 1 (cached if image exists)
RUN apt update              # Layer 2 (cached if layer 1 unchanged)
RUN apt install -y curl jq  # Layer 3 (cached if layer 2 unchanged)
WORKDIR /app                # Layer 4 (cached if layer 3 unchanged)
COPY . .                    # Layer 5 (rebuilds if ANY file chaneged)
CMD ["./start"]             # Layer 6 (rebuilds layer 5 chaneged)
```

### Cache invalidation

When a layer changes, all subsequent layers rebuild
```dockerfile
# Bad: any file changes reinstalls packages
COPY . .
RUN apt update && apt install -y curl jq

# Good: packages install early, only reinstalls when docker file changes
RUN apt update && apt install -y curl jq
COPY . .
```

>_"Order your `Dockerfile` from least frequently changes to most frequently changed"_

### Viewing layers

```bash
docker history myapp:1.0.0
```

## The build context

When we run `docker build`, docker actually sends the **build context** (the directory we specify) to the daemon.

```bash
docker build -t myapp:1.0.0 .
#                           ^ build context is current directory
```

### .dockerignore

Exclude files/directories from the build context with `.dockerignore`
```dockerignore
.git/
*.log
.env
Dockerfile
*.swp
*~
```

Benefits:
- faster builds (less data to send to the daemon)
- smaller images (exclude files that can/t be `COPY`ed)
- security (don't include secrets)

## Practical Example (Bash application with dependencies)

```dockerfile
FROM ubuntu:24.04

# Install dependencies
RUN apt-get update && \
    apt-get install -y --no-install-recommends curl jq && \
    rm -rf /var/lib/apt/lists/*

WORKDIR /app

# Copy scripts
COPY start process .
RUN chmod +x start process

CMD ["./start"]
```

This pattern works for most shell-based applications:
1. Start with Ubuntu (or another base)
2. Install system packages you need
3. Copy your scripts
4. Make them executable
5. Set the default command

## Building and Tagging

```bash
docker build -t myapp .                          # basic build
docker build -t myapp:1.0.0 .                    # build with specific tag
docker build -t myapp: latest .                  # build with latest tag
docker build -t myapp:1.0.0 -t myapp: latest .   # build with multiple tags
docker build -f Dockerfile.prod -t myapp:prod .  # specify dockerfile location
```

### Build arguments

Pass variables at build time
```dockerfile
ARG UBUNTU_VERSION=24.04       # default value if not mpassed through cli
FROM ubuntu:${UBUNTU_VERSION}

ARG APP_VERSION=1.0.0          # default value if not mpassed through cli
ENV APP_VERSION=${APP_VERSION}
```

```bash
docker build \
	--build-arg UBUNTU_VERSION=22.04 \
	--build-arg APPVERSION=2.0.0 \
	-t myapp .
```

### No cache

Force rebuild all layers
```bash
docker build --no-cache -t  myapp .
```

## Tagging best practices

Use semantic versioning
```bash
docker build -t myapp:1.0.0 .  # Points to exactly 1.0.0
docker build -t myapp:1.0 .    # Points to latest 1.0.x
docker build -t myapp:1 .      # Points to latest 1.x.x
```

Include git commit
```bash
docker build -t myapp:$(git rev-parse --short HEAD) .
```

 Never rely on latest
 ```bash
 # Bad
 docker build -t myapp:latest
 
 # Good
 docker build -t myapp:1.2.3
 ```

Tag for registries
```bash
docker build -t username/myapp:1.0.0              # for docker hub
docker build -t ghcr.io/username/myapp:1.0.0               # for github's ghcr.io
docker build -t registry.example.com/myapp:1.0.0  # for other registry
```

## Pushing to Github Container Registry

Github container registry (ghcr.io) lets you store container images alongside code

#### Create a PAT:
1. Github -> Settings -> Developer settings -> Personal access tokens -> Tokens (classic)
2. Click "Generate new token (classic)"
3. Select scopes \[`write:packages`, `delete:packages`\]
4. Copy token

#### Login to ghcr.io
```bash
export CR_PAT=ghp_xxxxxxxxxxxxxxxxxxxxxxxxxx  # store token (never commit)
echo ${CR_PAT} | docker login ghcr.io -u lenaggar --password-stdin  # login
```

#### Push image

Tag local image then push
```bash
docker tag backup:1.0.0 ghcr.io/lenaggar/backup:1.0.0
docker push ghcr.io/lenaggar/backup:1.0.0
```

Build local image then push
```bash
docker build -t ghcr.io/lenaggar/backup:1.0.1 .
docker push ghcr.io/lenaggar/backup:1.0.1
```
