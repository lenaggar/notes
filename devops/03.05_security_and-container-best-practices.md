## Why container security matters

Containers provide isolation but they are not magic:
- Containers share host kernel
- A vulnerability in the image is a vulnerability in production
- Mis-configured containers can expose the host
- Running as root in a container is dangerous

>_"Just because it is in a container doesn't mean it is secure.
>Defense in depth still applies."_

Two areas to focus on:
1. **Image security**: What goes in the image
2. **Runtime security**: How containers are run

## Running as non-root

By default containers run as root!
If an attacker escapes the container, they're root on the host (in some configurations)
```bash
docker run nginx:1.28  # inside the container, nginx master process is root
```

Solutions
1. use official images that include a non-root user
```dockerfile
FROM nginxinc/nginx-unprivileged:1.28  # runs as non-root (UID 101)
```

2. use `USER` instruction
```dockerfile
FROM ubuntu:24.04
RUN apt-get update && \
	apt-get install -y nginx && \
    rm -rf /var/lib/apt/lists/*
RUN useradd --create-home appuser  # Create non-root user
USER appuser                       # Switch to non-root user
CMD ["nginx", "-g", "daemon off;"]
```

Example
```bash
# Run default nginx (runs as root)
docker run -d --name root-nginx nginx:1.28
docker exec root-nginx whoami  # Output: root
docker rm -f root-nginx
```
The official nginx image runs as root by default. we can't add `--user nginx` because the nginx user lacks permissions for cache directories

```bash
# Run unprivileged nginx (runs as non-root by default)
docker run -d --name safe-nginx nginxinc/nginx-unprivileged:1.28
docker exec safe-nginx whoami  # Output: nginx
docker rm -f safe-nginx
```
Instead, use the official unprivileged nginx image. the `nginxinc/nginx-unprivileged` image is pre-configured to run as UID 101 (nginx user) with proper permissions for the image to function

>_"Use unprivileged base images when available.
>Build your own non-root setup when needed"_

## Multi-stage builds

Multi-stage builds produce smaller, more secure images by separating build-time dependencies from runtime

```dockerfile
# Stage 1: Build
From ubuntu:24.04 AS BUILDER
# ... instlla build tools , generate files

# Stage 2: Runtime
From nginx:1.28
COPY --from=builder /build/output /usr/share/ngnix/html
```

#### Example 
```bash:generate-status
#!/bin/bash

cat << EOF
<!DOCTYPE html>
<html>
<head><title>System Status</title></head>
<body>
<h1>System Status</h1>
<p>Generated: $(date)</p>
<p>Hostname: $(hostname)</p>
</body>
</html>
EOF
```

```dockerfile
# Stage 1: Generate html
FROM ubuntu:24.04 AS builder
WORKDIR /build
COPY generate-status .
RUN chmod +x generate-status && \
	./generate-status > index.html
	
# Stage 2: Serve with nginx (unpriveleged)
FROM nginxinc/nginx-unprivileged:1.28
COPY --from=builder /build/index.html /usr/share/nginx/html/index.html
EXPOSE 8080
```

```bash
docker build -t status-page:1.0.0
docker run -d -p 8080:8080 --name status-page status-page:1.0.0
curl localhost:8080
ssl -L 8080:localhost
docker rm -f status-page
```

The final image is much smaller, does not contain `bash`, nor `generate-status` build script
This matters because many devops tools like `kubectl`, `docker`, `terraform` use multi-stage builds with GO to produce tiny images.

##### Benefits
| Benefit               | Explanation                          |
| --------------------- | ------------------------------------ |
| Smaller images        | Only runtime dependencies included   |
| Faster deployments    | Less data to transfer                |
| Smallr attack surface | No build tools in production         |
| Cleaner separation    | Build concerns separate from runtime |

>_"Separate what we need to **build** from what we need to **run**"_

#### Port forwarding

If the container is running on a remote server, we can access the html page on our browser by doing this simple yet powerful trick:

```bash
ssh -L 7070:localhost:8080 example.com
```
Basically, what we are doing here:
1. we are telling the ssh client that we want to connect to a remote server `example.com` (it could be an IP address for a machine on a server on a local network)
2. and while we are connecting, we want to build a tunnel where connections sent to our local TCP port (`7070` | `localhost:7070`) on the local (client) host, are to be forwarded through the ssh connection to the given host and port on the remote side's `localhost:8080` (server)
Format:
```bash
ssh -L [bind address:]port:host:hostport
ssh -L [bind address:]port:remote_socket
ssh -L local_socket:host:hostport
ssh -L local_socket:remote_socket
```

## Choosing base images

Base images should be chosen wisely!

| Image                | Size   | Use Case                     |
| -------------------- | ------ | ---------------------------- |
| nginx:1.28           | ~187MB | Web server, ready to use     |
| nginx:1.28-alpine    | ~43MB  | Smaller nginx                |
| ubuntu:24.04         | ~77MB  | Familiar, full featured      |
| debian:bookworm-slim | ~74MB  | Smaller Debian, good balance |
| alpine:3.20          | ~7MB   | Tiny, uses `musl` `libc`     |

>_"If you can use alpine for your app,
>then use `alpine` because it is the most minimal and hence the most secure as well"_

Nginx variants
```bash
docker pull nginx:1.28         # standard (debian-based)
docker pull nginx:1.28-alpine  # alpine-based (smaller)
```

Alpine Linux
```dockerfile
FROM alpine:3.20
RUN apk add --no-cache bash curl
```
Pros:
- Tiny (5~7MBs)
- Has `apk` package manager
- Good enough foe many usecases
Cons:
- Uses `musl`, `libc` (not `glibc`) - some compatibility issues
- Some packages unavailable or behave differently
- Debugging can be harder (different tools)

## Best practices checklist

### Image security
- [ ] Use specific image tags, not `latest`
- [ ] Run as non-root
- [ ] Use multi-stage builds to minimize image size
- [ ] Choose minimal base images (`slim`, `alpine`, `distroless`) when possible
- [ ] No unnecessary packages
- [ ] Scan images for vulnerability

### Dockerfile practices
- [ ] Pin versions for base images and packages
- [ ] Order instructions from least to most frequently changed
- [ ] Combine `RUN` commands to reduce layers
- [ ] Use `.dockerignore` to exclude unnecessary files from build context
- [ ] Don't store secrets in images (inject at runtime)
- [ ] Add `HEALTHCHECK` for production images

### Never include in images
- [ ] API keys or passwords
- [ ] Private SSH keys
- [ ] `.env` files with secrets
- [ ] AWS credentials
- [ ] Database connection strings with passwords

> _"Treat your image as public.
> If it leaked, would you be compromised?"_

## Health checks

Health checks let docker know if your container is healthy (Essential for production env)

Adding a health check to `nginx`
```dockerfile
FROM nginx:1.28
RUN apt update && \
	apt install -y --no-install-recommends curl && \  # for healthchecks
	rm -rf /var/lib/apt/lists/*
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
	CMD curl -f http://localhost/ || exit 1
EXPOSE 80
```
Options:
- `--interval`: time between checks (defaults 30s)
- `--timeout`: max time for check to complete
- `--start-period`: grace period
- `--retries`: failures before making container unhealthy

```bash
docker build -t healthy-nginx:1.0.0 .                    # build
docker run -d -p 8080:80 --name web healthy-nginx:1.0.0  # run
docker ps                                                # check health status
docker inspect --format='{{.State.Health.Status}}' web   # detailed health info
docker rm -f web                                         # cleanup
```

## Putting it all together

#### Step 1: Create `generate-page`
```bash
#!/bin/bash
# Generate a status page with system info

cat << EOF
<!DOCTYPE html>
<html>
<head>
  <meta charset="UTF-8">
  <title>Server Status</title>
  <style>
    body { font-family: sans-serif; margin: 40px; }
    .status { color: green; }
  </style>
</head>
<body>
  <h1>Server Status</h1>
  <p class="status">Online</p>
  <p>Built: $(date)</p>
  <p>Build Host: $(hostname)</p>
</body>
</html>
EOF
```

#### Step 2: Create `Dockerfile`
```dockerfile
# Stage 1: generate static html
FROM ubuntu:24.04 AS builder
WORKDIR /build
COPY generate-page .
RUN chmod +x generate-page && ./generate-page > index.html

# Stage 2: production nginx (unprivileged to run as nono-root)
FROM nginxinc/nginx-unprivileged:1.28
LABEL org.opencontainers.image.source="https://github.com/user/repo"
LABEL org.opencontainers.image.versiojn="1.0.0"
COPY --from=builder /build/index.html /usr/share/nginx/html/index.html
EXPOSE 8080
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
	CMD curl -f http://localhost:8080/ || exit 1
```
The `nginxinc/nginx-unprivileged` image:
- runs as non root user `nginx` (UID 101) by default
- listens on port 8080 (*non-root can't bind to  ports below 1024*)
- has proper directory permissions pre-configured

#### Step 3: Create `.dockerignore` file
```dockerignore
.git
*.md
Dockerfile
.dockerignore
```

#### Step 4: Build and run
```bash
docker build -t status-server:1.0.0 .                         # build
docker run -d -p 8080:8080 --name status status-server:1.0.0  # run
curl localhost:8080                                           # test
docker ps                                                     # cheack health
docker exec status whoami                                     # verify non-root
docker rm -f status                                           # cleanup
```

#### What this image demonstrates

| Best practice     | How it is applied                                       |
| ----------------- | ------------------------------------------------------- |
| Multi-stage build | Generator runs in builder, only HTML goes to production |
| Non-root user     | Uses `nginxinc/nginx-unprivileged` (runs as UID 101)    |
| Health check      | Checks `localthost:8080` every 30 seconds               |
| Minimal image     | No `bash` or build tools in final image                 |
| Pinned versions   | `nginxinc/nginx-unprivileged:1.28`, `ubuntu:24.04`      |
| Labels            | OCI metadata included                                   |
