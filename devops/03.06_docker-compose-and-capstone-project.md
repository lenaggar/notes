We will be building a multi-container application using docker compose. The application is a dad-joke dashboard where one container fetches jokes and another serves them.

A live dashboard with two containers
```
┌─────────────────┐         ┌─────────────────┐
│     updater     │         │      nginx      │
│     (bash)      │         │                 │
│                 │         │                 │
│  fetch jokes    │         │   serve HTML    │
│  write HTML ────┼────────▶│                 │
└─────────────────┘         └─────────────────┘
         │                          │
         └──────── /html ───────────┘
              (shared volume)
```
- updater: a bash container that fetches dad jokes from an api every 30 seconds
- nginx: serves the generated HTML page
- shared volume: both containers access the same `/html` directory

---
## Why docker compose

So far we have run single containers but real applications often need multiple containers working together:
- web server + database
- api + cache + worker
- frontend + backend + message-queue

Managing these with individual `docker run` commands gets painful
```bash
# without compose, it becomes messy very quickly
docker network create myapp
docker volume create mydata
docker run -d --name db --network myapp -v mydata:/var/lib/postgresql/data postgres:16
docker run -d --name api --network myapp -e DATABASE_URL=postgres://db/app myapp:1.0.0
docker run -d --name web --network myapp -p 80:80 nginx:1.28
```

Docker compose lets us define everything in one file:
```yaml
services:
	db:
		image: postgres:16
		volumes:
			- mydata:/var/lib/postgresql/data
	api:
		image: myapp:1.0.0
		environment:
			- DATABASE_URL=postgres://db/app
	web:
		image: nginx:1.28
		ports:
			- "80:80"

volumes:
	mydata:
```
And start everything with one command `docker compose up`

## Updater script

Create `updater`
```bash:updater
#!/bin/bash
# Fetch dad jokes and update the HTML page

API_URL="https://icanhazdadjoke.com/"
HTML_FILE="/html/index.html"

echo "Starting joke updater..."

while true; do
  # Fetch a random dad joke
  JOKE=$(curl -s "$API_URL" -H "Accept: text/plain")

  # Generate HTML
  cat > "$HTML_FILE" << EOF
<!DOCTYPE html>
<html>
<head>
  <title>Dad Joke Dashboard</title>
  <style>
    body {
      font-family: sans-serif;
      max-width: 600px;
      margin: 50px auto;
      padding: 20px;
      text-align: center;
    }
    .joke {
      font-size: 1.5em;
      margin: 40px 0;
      padding: 20px;
      background: #f0f0f0;
      border-radius: 10px;
    }
    .meta {
      color: #666;
      font-size: 0.9em;
    }
  </style>
</head>
<body>
  <h1>Dad Joke Dashboard</h1>
  <div class="joke">$JOKE</div>
  <p class="meta">Updated: $(date)</p>
  <p class="meta">Refreshes every 30 seconds</p>
</body>
</html>
EOF

  echo "Updated joke: $JOKE"
  sleep 30
done
```

## Updater Dockerfile

Create `Dockerfile.updater`:
```dockerfile:Dockerfile.updater
FROM ubuntu:24.04
Run apt-get update && \
	apt-get install -y --no-install-recommends curl ca-certificates && \
	re -rf /var/lib/apt/lists/*
WORKDIR /app
COPY updater .
RUN chmod +x updater
CMD ["./updater"]
```

---
## Docker compose file

Create `docker-compose.yml`:
```yaml:docker-compose.yaml
services:
	updater:
		build:
			context: .
			dockerfile: Dockerfile.updater
		volumes:
			- html:/html
		restart: unless-stopped
	web:
		image: nginx:1.28
		ports:
			- "8080:80"
		volumes:
			- html:/usr/share/nginx/html:ro
		restart: unless-stopped
	
volumes:
	html:
```


## How `docker compose` works

1. **Docker compose creates a network**: both containers can talk to each other by service name
2. **Creates a shared volume**: `html` volume mounted in both containers
3. **Starts updater**: fetches jokes, writes HTML to volume
4. **Starts nginx**: served html from the same volume
5. **Every 30 seconds**: updated fetches a new joke, overwrites HTML
6. **Browser refresh**: shows a new joke

And the shared volume connection works as follows:
```yaml
# updater writes to shared volume here
volumes:
	- html:/html

# nginx reads from shared volume here
volumes:
	- html: /usr/share/nginx/html:ro  # <- (:ro) means nginx mounts it readonly
```
Same volume `html` mounted twice at the same time (different mount paths)

## Apply best practice: Running as non-root

The simple version above runs both containers as root. Let's make both of them run as non-root to limit the impact of security breaches.

### The challenge

Running as non-root with shared volume is tricky:
- Docker volumes are created with root ownership
- Non-root users can't write to root-owned directories
- Both containers need matching user IDs to share files

### The solution: init container pattern

Use a one-shot container to fix the permissions before the main containers start, this means that the `docker-compose.yaml` should be:
```yaml:docker-compose.yaml
services:
	init:
		image: busybox
		command: chown -R 101:101 /html
		volumes:
			- html:/html
	updater:
		build:
			context: .
			dockerfile: Dockerfile.updater
		user: "101:101"  # <- non-root user
		volumes:
			- html:/html
		restart: unless-stopped
		depends_on:  # <- added dependency
			init:
				condition: service_completed_successfully
	web:
		image: nginxinc/nginx-unprivileged:1.28  # <- unprivileged
		ports:
			- "8080:80"
		volumes:
			- html:/usr/share/nginx/html:ro
		restart: unless-stopped
		depends_on:  # <- added dependency
			init:
				condition: service_completed_successfully

volumes:
	html:
```

The `Dockerfile.updater` also needs updates to follow best practices:
- multi-stage build, and
- non-root user

```dockerfile:Dockerfile.updater
# stage 1: build
FROM ubuntu:24.04 AS builder
WORKDIR /build
COPY updater .
RUN chmod +x updater

# stage 2: runtime
FROM alpine:3.21
RUN apk add --no-cache curl ca-certificates bash
RUN addgroup -g 101 appgroup && \
	adduser -u 101 -G appgroup -D appuser
WORKDIR /app
COPY --from=builder --chown=appuser:appgroup /build/updater .
USER appuser
CMD ["./updater"]
```

## This all leads to Kubernetes

Docker compose is great for:
- local development
- simple deployments
- single-shot applications

But it can't:
- scale across multiple servers
- automatically restart failed containers on other hosts
- handle rolling updates without downtime
- manage secrets securely
- load balance traffic

That's what Kubernetes does. The concepts we learned here (services, volumes, networking) translate directly to Kubernetes.
