## How linux enables containers
Containers aren't magic, they are built on linux kernel features

### Namespaces (isolation)
Namespaces make a process think it is alone on the system

| Namespace | What  it isolates                            |
| --------- | -------------------------------------------- |
| PID       | process IDs - container sees itself as pid#1 |
| NETWORK   | network interfaces, ports, routing           |
| MOUNT     | filesystem mount points                      |
| USER      | use and group ids                            |
| UTS       | hostname                                     |
A container process might be pid#1 inside the container but pid#4827 on the host. It can't see other processes

Here's how to start a linux namespace
```bash
#!/bin/bash

sudo unshare --fork --pid --mount-proc bash
```
Let's describe what happens here ^
- `unshare` => command that creates a new namespace specified with the cli options provided (if no program is provided, the `${SHELL}` is run)
- `--fork` => fork the specified program (`bash` in this case) as a child process of `unshare` rather than running it directly
- `--pid` => creates a new `PID` namespace
- `--mount-proc` => mount the proc to filesystem at provided mountpoint (if not provided, defaults to `proc`) useful when creating a new `PID` namespace. also implies creating a new `MOUNT` namespace

After running this ^ command, we'll get a new prompt showing that we are running as `root` this is a new `namespace` that we are currently inside of.
Now try run `pstree -p` to see the process tree in that new namespace, what we'll get is  something like this:
```
[root@naggar1 mahmoud]# pstree -p
bash(1)───pstree(3)
```

This namespace that we created _thinks_ it is the only thing that is running on the system!

In essence, this is a self contained linux environment that shares the host system kernel, but it has no clue what other processes exist outside of its namespace. I sees itself as pid#1.

The problem is that this new `namespace` still has access to all the host system resources such as available memory and cpu. This is where `cgroups` (control groups) can be used to set resource limits the processes inside a namespace.

### Cgroups (control/resource limits)
Control groups limit what host system resources a container can use:
- CPU: _"This Container gets max 50% of one core"_
- Memory: _"This container gets max 512MBs"_
- Disk I/O: _"This containe gets max 100MB/s read"_
Without `cgroups`, one container could starve other processes for resources.

Create a `cgroup`
```bash
sudo mkdir /sys/fs/cgroup/nigo
```

Set a 10MB memory limit
```bash
echo 10000000 | sudo tee /sys/fs/cgroup/nigo/memory.max
```

Add current shell to the group
```bash
echo $$ | sudo tee /sys/fs/cgroup/nigo/cgroup.procs
```

Now try to use 15MB of memory
```bash
head -c 15MB /dev/zero | tail
```
The command should fail with "Killed" message because we are trying to write bytes to memory that exceed the maximum limit that we set for the process pid

This is fundamentally how containers work!

### Union Filesystem (efficiency)
Images are built in layers. Union filesystems (like OverlayFS) stack these layers.

#### OverlayFS Primer ([source](https://grant.pizza/blog/overlayfs/))
Overlay filesystems allow you to take multiple directory trees and view them as if they were _merged_ together.

So, imagine we have these three separate directories and we want to _merge_ them together. Each contain various files. Notice that `/tmp/midle` and `/tmp/upper`contain files wioth the same name `c.txt`:
![[overlayfs-1.png]]
Let's say we want to create an overlayfs with these three directories. We can achieve that with this command
```bash
mount \
    -t overlay \
    -o \
    lowerdir=/home/grant:/tmp/middle, \
    upperdir=/tmp/upper,\
    workdir=/tmp/workdir \
    none \
    /tmp/overlay
```
The order is important here ^ Different order can result in different outputs. The order in which we specify them is the order they are effectively stacked on top of one another.
![[overlayfs-2.png]]
After the stacking, here's what it looks like
![[overlayfs-3.png]]
In this case `c.txt` is the one from the upper filesystem.

Explanation from `man mount`
> An overlay filesystem combines two filesystems - an `upper filesystem` and a `lower filesystem`. When a  name  exists  in  both filesystems, the object in the `upper filesystem` is _visible_ while the object in the `lower filesystem` is either _hidden_ or, in the case of directories, _merged_ with the upper object.

All of the layers except the top one are read-only.

##### Benefits
Multiple containers can use the same base image. The root filesystem (`rootfs`) of the image is stored somewhere on a disk. An overlayfs is created for each running container where the bottom layer is the `rootfs`

One cool usecase is creating an overlay mount of home directory and mount it into a container so that the environment lookslike it is home,but any changes done inside the container won't persist once the container is destroyed. Useful for installing programs that may break dependencies.

---
#### Why layers matter?
- Shared layers are cached => downloaded once, used many times
- 10 containers from the same image share base layers
- Only differences are stored per container => size efficient

## What is Docker
Docker is a platform that makes Linux containers easy to use. It wraps those three Linux kernel features we discussed above (`namespaces`, `cgroups`, and `union filesystems`) into simple commands.

### Docker architecture
Docker is composed of two parts:
- docker client/CLI (aka `docker`) => the command we type
- docker server/daemon (aka `dockerd`) => the background service managing containers
- docker desktop => GUI \[+ Linux VM (for Mac/Windows)\]

Docker CLI talks to the docker daemon through an API. So, when we type `docker run` what's actually happening is that the CLI sends a request to the daemon, which does the actual work.

### Docker Desktop vs Docker Engine
Docker Desktop (Mac, Windows, Linux GUI):
- Includes everything: CLI, daemon, GUI, Kubernetes
- Runs Linux containers in a lightweight VM (for Mac, and Windows)
- Free for personal use, paid for large enterprises

Docker Engine (Linux only!):
- Includes: CLI, daemon
- Runs containers natively (no VM needed)
- Always free and open source
