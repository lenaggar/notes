## Run vs Exec vs Attach

1. `docker run` => Create a NEW container (`docker run -it ubuntu bash`)
	- creates a **new** container from an image
	- starts it with the specified command (`bash`)
	- every `run` creates a new container
2. `docker exec` => Run in EXISTING container (`docker exec -it mycontainer bash`)
	- runs a command in already running container
	- spawns a **new process** inside the container
	- container must e running (not stopped)
3. `docker attach` => Connect to EXISTING process (`docker attach mycontainer`)
	- connects to the **main process** (PID#1) of a running container
	- shares the same `stdin`/`stdout`
	- **dangerous**: running `ctrl+c` may stop the container
> _"Use exec to get a shell.
> Use attach only when need to interact with the main process"_

#### More on exec
Let's first have a running container
```bash
docker run -d --name webserver -p 8080:80 nginx
```

- `docker exec -it webserver bash` => bash (if available)
  `docker exec -it webserver sh` => sh (almost always available)
  `docker exec -it webserver /bin/ash` => specific shell (Alpine)
	- all of them gets you an interactive shell from a running container because of (`-it`)
- `docker exec webserver env` => check env
  `docker exec webserver la -la /app` => list files
  `docker exec webserver ps aux` => check processes
  `docker exec webserver ping -c 3 google.com` => test network
	- all of them run a single command in a running container and return the result
- `docker exec -u root wbserver whoami`
  `docker exec -u 1000 webserver id`
	- run as a different user
- `docker exec -e DEBUG=true webserver .my_script`
	- with environment variables

## Starting and Stopping
- **Stop (graceful)** => `docker stop webserver`
	1. sends `SIGTERM` to main process
	2. waits 10 seconds (configurable with `-t`)
	3. sends `SIGKILL` if still running
- **Kill (Immediate)** => `docker kill webserver`
	- sends `SIGKIL` immediately. when a container won't stop gracefully
- **Start (Stopped container)** => `docker start webserver`
	- starts a previously stopped container data in container layer persists
- **Restart** => `docker restart webserver`
	- equivalent to `stop` then `start`

Subtle nuance:
- `docker run` => creates a NEW container from an image
- `docker start` => starts an existing stopped container

## Restart policies
When a container crashes or the Docker daemon restarts; we can set a restart policy:
```bash
docker run -d --restart=always nginx
```

Options:
- `--restart=no` => never restart (default)
- `--restart=on-failure` => only if exit code is non-zero
- `--restart=on-failure:3` => only 3 restart attempts
- `--restart=always` => always restart including daemon restart
- `--restart=unless-stopped` => like `always` but not if manually stopped

Practical use cases:
- `docker run -d -restart=always --name db pstgres` => databases should always be running
- `docker run -d -restart=unless-stopped --name web nginx` => webserver should survive reboots
- `docker run -d -restart=on-failure:3 --name worker myworker` => one off job, attempt to retry 3 times on failure

Updating the restart policy:
- `docker update --restrt=always webserver` => updates an existing container's restart policy

>_"Use `unless-stopped` for services,
>It restarts on crashes and reboot but also respect manual stops"_

## Inspecting containers
We can inspect a running container and get a JSON with everything: network, mounts config, state, etc...
```bash
docker inspect -f 
```

We can also inspect for specific fields by providing a `--format` (or `-f`) with a [go template](https://docs.docker.com/engine/cli/formatting/) like so:
- `docker inspect -f '{{.NetworkSettings.IPAddress}}' webserver` => get container's IP address
- `docker inspect -f '{{.State.Status}}' webserver` => get container status
- `docker inspect -f '{{.Config.Env}}' webserver` => get environment variables

## Resource usage
We can also view docker's resource usage
```bash
docker stats
```

- `docker stats` => returns a live stats view
- `docker stats --no-stream` => returns only a snapshot of the current resource usage

## Volumes for persistent data
It is always important to remember that a container's filesystem is ephemeral, when the container is `deleted` its data is lost

The problem:
```bash
# Run a container, create a file
docker run -d --name mybox alpine sleep 3600
docker exec mybox sh -c "echo 'important data' > /tmp/myfile.txt"
docker exec mybox cat /tmp/myfile.txt  # shows: important data

# Remove and recreate
docker rm -f mybox
docker run -d --name mybox alpine sleep 3600
docker exec mybox cat /tmp/myfile.txt  # errors: no such file
```

Container's shouldn't persist data inside them, they should be stateless! If we want to persist data, we should use `volumes`.

The solution: Volumes
```bash
# Create a named volume
docker volume create mydata

# Use it, this mounts the created volume to /data inside the container
docker run -d \
  --name mybox \
  -v mydata:/data \
  alpine sleep 3600
  
# Create a file from the mounted volume
docker exec mybox sh -c "echo 'important data' > /data/myfile.txt"

# Remove the container, then recreate it
docker rm -f mybox
docker run -d \
  --name mybox \
  -v mydata:/data \
  alpine sleep 3600

docker exec mybox cat /data/myfile.txt  # still hows: important data
```

Data persists because it lives outside the container on the host.
What `-v VOLUME_NAME:CONTAINER_MOUNT_POINT` did is that it mounts the volume to the container's specified mount point and the container has access to read form/write to it.

But to be more precise:
- The container has its own _writable_ layer (ephemeral)
- When we mount the volume (`-v mydata:/data`) to it, this is what happens:
	- `/data` is **no longer part of the container filesystem**
	- It is **a mount point backed by the volume**
- So:
	- data written to `/data` goes to the **volume** (persists)
	- data written elsewhere stays in the **container layer** (lost on container removal)

By default, Docker stores and manages the volumes on the host filesystem:
- `/var/lib/docker-data/volumes/` => where volumes that docker manages exist
- `/var/lib/docker-data/volumes/mydata/_data/` => where actual contents of the volume exist

To check where exactly data inside a volume is mounted to on the host:
- `docker volume inspect` => shows information about the volume including the `MountPoint` on the host device

Volume commands:
- `docker volume ls` => list volumes
- `docker volume inspect mydata` => inspect a volume like we saw above
- `docker volume rm mydata` => remove a volume
- `docker volume prune` => removed unused volumes

### Bind Mounts (Host Directory)
If we want to mount a directory on the host (not a volume) into a container
```bash
docker run -it \
  -v $(pwd):/app \
  --name dev \
  ubuntu:24.04 bash
```
So, what happens when we do that ^
- Changes on the host appear in container, and vice versa
- Useful for local development (edit on host, run in container)

A use case for example here would be that I am building a bash script and I want to make sure that it will work on multiple distros, so I try the source script/program I have on the host, bind mount it into different containers each based on different Linux distros, and test i out. Pretty cool!

### Volumes vs Bind Mount
| Feature    | Volume                   | Bind Mount             |
| ---------- | ------------------------ | ---------------------- |
| Managed by | Docker                   | You                    |
| Location   | Docker's data directory  | Anywhere on host       |
| Backup     | `docker volume` commands | Normal files and tools |
| Use case   | Persistent data (DBs)    | Development, config    |

## Cleanup commands

Remove stopped containers:
- `docker rm webserver` => remove single stopped container
- `docker container prune` => remove all stopped containers

Remove images:
- `docker rmi nginx:1.28` => remove single image
- `docker image prune` => remove unused images
- `docker image prune -a` => remove all unused images (not just dangling)

Remove volumes:
- `docker volume rm pgdata` => remove single volume
- `docker volume prune` => remove unused volumes

Remove everything:
- `docker system prune` => remove all unused data (careful!)
- `docker system prune -a --volumes` => remove all unused data including volumes and all images (very careful!)

Check disk usage:
- `docker system df` => shows an overview of the images, containers, and volumes available including their number, their size, and reclaimable size
