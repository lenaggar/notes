- `k get pods` => gets a list of minimal info about running pods
- `k get pod enginx` => gets minimal info about running pod
- `k get pod enginx -o yaml` => gets the manifest of running pod
- `k edit pod nginx` => opens an editor with the pod manifest to update 
					  (NOT RECOMMENDED)
- `k describe pod nginx` => see the updates done through the edit command above

If we want to create a kubernetes pod from a yaml file, we need a yaml file to get started. some of the ways we can create a pod yaml file:
1. look up on the internet to get some yaml
2. copy/paste from somewhere else to have a structure that we can start with then modify and extend from it (this is the most used way in production systems)
3. generating yaml from the command line (let's do that)

- `k run nginx-yaml --image=nginx --dry-run=client -o yaml` => this outputs the pod configuration as yaml file without actually running the pod
	- `--dry-run=client` => print the object that the `kubectl` client would send to the api-server without sending it (print the corresponding API objects without creating them)
	- `--dry-run=server` => test the object that the `kubectl` client would send to the api-server and delete it right after (this is way for example to test if the creation process of a pod will take place without resulting in any error)
	- `-o yaml` => output format in yaml
- `k run nginx-yaml --image=nginx --dry-run=client -o yaml > nginx.yaml`

After we are done with editing the output of the nginx dry run outputted yaml pod specification file, now we can create a pod from that newly created manifest.
We can do that via:
- `k create -f nginx.yaml` => only create the pod
- `k apply -f nginx.yaml` => create the pod **and** detect the change between a running pod (if any) and the pod's specification files, the output will be:
	- ` pod/nginx-yaml created` => if there was no pod running
	- ` pod/nginx-yaml configured` => if there was a pod running but specs changed
	- ` pod/nginx-yaml unchanged` => if there was a pod running but specs not changed
