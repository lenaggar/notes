- `k create deployment test --image=httpd -- replicas=3` => creates a deployment based on `httpd` image with 3 replicas

We can also output the yaml from such command to have as a starting point then modify on top to get to our desired end state:
```bash
k create deployment <deployment-name> \
	--image=httpd --replicas=10 \
	-o yaml > deploy.yaml
```

#### So, What is a `Deployment`?

A `Deployment` is a declarative way to create/update pods and `ReplicaSets`. We describe the desired end state in a deployment, and the deployment controller changes the actual state to match the desired state "at a controlled rate"

#### Then, What is a `ReplicaSet`?

A `ReplicaSet`'s purpose is to maintain a stable set of replica pods at any given time. It is usually managed by a `Deployment` that we create.

Every `Deployment` has a `Strategy` that controls the way Kubernetes should update the existing pods. We have `"Recreate" | "RollingUpdate"`:
- `.spec.strategy.type==Recreate` => all existing pods are killed before the new ones are created
- `.spec.strategy.type==RollingUpdate` => gradually scale down the old ReplicaSets and scale up the new one

In `RollingUpdate` strategy also, we can set two optional parameters `maxUnavailable`, and `maxSurge`:
- `maxUnavailable` => can be an absolute number or a percentage, it represents the acceptable number of pods that can be out of service during a rolling update
- `maxSurge` => can be an absolute number or a percentage, it represents the maximum number of pods that can be worming up beside the running instances to update them

>_"`surge` costs money (extra infra),
>`unavailability` costs users (fewer unhealthy instances)"_

Also in Kubernetes, when a new deployment is failing for some reason, the update rollout will stop at the first instances that were used for the rollout
