### Pods Networking
Networking in kubernetes also happen on the pod level.
Meaning that kubernetes is not connecting containers to each other, it is connecting pods to each other.
In the cluster, each pod gets its own unique cluster-wide IP address.

When we setup a cluster from scratch using `kueadm` or `talos` for example, we can configure a `CIDR` range which is the IP address pool that the cluster will use to assign to pods.
By default, pods can connect to all pods on all nodes, on all namespaces.
However, that can be limited for a pod using `network policies`, so for example we can configure that pods in the `mealie` namespace can only interact with other pods on the `mealie` namespace.
Containers in pods can communicate with each other through localhost. Since they all share the same IP address, however, each container can have its own port number (mental model: a pod is like a small vm that containers run inside as processes).
>_"A pod has its own private network namespace which is shared by all of the containers within the pod"_

### CNI Plugin
So, how does kuernetes do this? connecting pods with each other? this is done through `CNI Plugin`(Container Networking Interface).
CNI provides newtwork connectivity to containers.
CNI configures network interfaces in containers (such as hardware NICs).
CNI assigns IP addresses to these network interfaces in containers & also setup the routes.
In Linux networking we have `iptables`, so every node in the cluster is a virtual machine running linux and Kubernetes' CNI configures the network on that node through `iptables`.

When setting up the cluster from scratch we need to select the CNI plugin to use for cluster networking, examples:
- cilium
- calico
- flannel (basic, Rancher Desktop & k3s use it)

### Services

This is the main concept that we need to grasp in order to get good at Kubernetes networking.

A service offers a _consistent address_ to access a set of pods. So it is a grouping of pods essentially.

Why do we need services?
- Pods are *ephemeral*. We should not expect a pod to have a long lifespan.
- Pods are constantly changing and being moved across nodes.
- How will the System keep track of the constantly changing IP address?

```
Service A (Frontend Service for example)
  |___ Pod 1
  |___ Pod 2
  |___ Pod 3
  .

Service B (Backend Service for example)
  |___ Pod 4
  |___ Pod 5
  .
```

When using Services, we don't really care which pod that it is using is handling our request, the Service is handling that for us. So what need to do is to point our application to the Service and then Kubernetes will handle the rest of it.
If a pod is updated (rolling update for example), removed, added to the service, we don't really care, we only care about the service from a cluster networking perspective such as routing.

Let's inspect the existing services before we create anything:
```bash
$ k get service
NAME         TYPE        CLUSTER-IP   EXTERNAL-IP   PORT(S)   AGE
kubernetes   ClusterIP   10.43.0.1    <none>        443/TCP   11d
```

Now let's create a new quick deployment called `frontend`:
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  labels:
    app: frontend  # <-- notice the selector here
  name: frontend
spec:
  replicas: 10
  selector:
    matchLabels:
      app: frontend
  template:
    metadata:
      labels:
        app: frontend
    spec:
      containers:
        - image: httpd:alpine3.18
          name: httpd
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1
      maxSurge: 1
```

And now, let's apply the deployment:
```bash
$ k apply -f frontend.yaml
```

When exposing a group of pods as a service, we get a service object that represents the service.
The service has its own unique "Cluster IP" address which will remain the same always no matter how much the pods it groups change.
The service will also have its own unique name that we can use to point to it anywhere from our cluster and the name will be DNS-resolved into the service Cluster IP address.
The service would use selectors that target the pods it is grouping.
The service will also load-balance the traffic between the pods for us out of the box.

Services have different type:
1. ClusterIP (default type):
	   Creates a cluster-wide IP address for the service. As long as the service exists, that unique cluster IP won't change.

2. NodePort:
	   Exposes a port on each node allowing direct access to the service through any node's IP address.
	   Since the nodes are VMs, those VMs can expose ports.
	   - NodePorts are not used that much in production environments.

3. LoadBalancer:
	   Used for cloud providers.
	   If we create an AKS cluster for example on Azure, this will create an Azure LoadBalancer to route traffic into the cluster.
	   Can also be used in (k3s or rd)

When we created the `frontend` service, Kubernetes set the type as the default value `ClusterIP`. We can change that to `LoadBalancer` and see how does that change it:

Let's edit the service manifest using `k edit service frontend`.

This is how the `ClusterIP` service manifest looks like:
```yaml

# and an empty file will abort the edit. If an error occurs while saving this file will be
# reopened with the relevant failures.
#
apiVersion: v1
kind: Service
metadata:
  creationTimestamp: "2026-05-22T22:25:30Z"
  labels:
    app: frontend
  name: frontend
  namespace: default
  resourceVersion: "296971"
  uid: 1a16212b-d753-42ae-a248-ac39a977836e
spec:
  clusterIP: 10.43.42.26
  clusterIPs:
  - 10.43.42.26
  internalTrafficPolicy: Cluster
  ipFamilies:
  - IPv4
  ipFamilyPolicy: SingleStack
  ports:
  - port: 8080
    protocol: TCP
    targetPort: 8080
  selector:
    app: frontend
  sessionAffinity: None
  type: ClusterIP   # <-- type is ClusterIP
status:
  loadBalancer: {}  # <-- empty for ClusterIP
```

This is how the `LoadBalancer` service manifest looks like:
```yaml
# Please edit the object below. Lines beginning with a '#' will be ignored,
# and an empty file will abort the edit. If an error occurs while saving this file will be
# reopened with the relevant failures.
#
apiVersion: v1
kind: Service
metadata:
  creationTimestamp: "2026-05-22T22:25:30Z"
  finalizers:
  - service.kubernetes.io/load-balancer-cleanup  # <-- new
  labels:
    app: frontend
  name: frontend
  namespace: default
  resourceVersion: "297024"
  uid: 1a16212b-d753-42ae-a248-ac39a977836e
spec:
  allocateLoadBalancerNodePorts: true  # <-- new
  clusterIP: 10.43.42.26
  clusterIPs:
  - 10.43.42.26
  externalTrafficPolicy: Cluster  # <-- new
  internalTrafficPolicy: Cluster
  ipFamilies:
  - IPv4
  ipFamilyPolicy: SingleStack
  ports:
  - nodePort: 32244  # <-- new
    port: 8080
    protocol: TCP
    targetPort: 8080
  selector:
    app: frontend
  sessionAffinity: None
  type: LoadBalancer  # <-- type is LoadBalancer
status:
  loadBalancer:
    ingress:          # <-- populated for LoadBalancer
    - ip: 192.168.178.100
      ipMode: VIP
```

```bash
$ k get svc
```

```yaml
    app: mealie
  name: mealie
  namespace: mealie
spec:
  ports:
  - port: 9000
    protocol: TCP
    targetPort: 9000
  selector:
    app: mealie
  type: LoadBalancer
```

### Ingress

Ingress is a resource on the cluster and it exposes HTTP/HTTPS routes from outside the cluster to services within the cluster.
So the cluster is listening to a certain domain and it has a router configured to route that domain to a certain service inside the cluster.

Features:
- SSL/TLS Termination
- External URLs
- Path based routing

Ingress is implemented by a `yaml` ingress resource, just like the service resource.
So, we can create a new ingress by running `k create ingress`.

Ingress features are made possible by the Ingress Controller. Some of the common ones:
- nginx
- Traefik (used in rd and k3s)
- Cilium
- Cloud: (AGIC - Ingress controller in Azure cloude)
