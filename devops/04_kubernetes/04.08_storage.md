
In Containers, in order for the container to save its data somewhere, it needs to have a `volume` which is like a virtual disk mounted to it. In kubernetes, it is exactly the same.

>_"Kubernetes volumes provide a way for containers in a Pod to access and share data via the filesystem"_

For containers in a pod to use a volume:
1. we specify specify the volumes to provide for the Pod
   in `.spec.volumes`, and
2. we declare where to mount those volumes into containers
   in `.spec.containers[*].volumeMounts`.

Let's check this simple `pod` yaml file for example:
```yaml:nginx-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  labels:
  name: nginx-storage
spec:
  containers:
    - image: nginx
      name: nginx
      volumeMounts:
        - mountPath: /scratch
          name: scratch-volume
  volumes:
    - name: scratch-volume
      emptyDir:
        sizeLimit: 500Mi
```

Here we see two things: `volumes` and `volumeMounts`:
- `volumes` are defined on the pod level, these basically state what volumes are available for this pod.
  So, `scratch-volume` is the volume that is being attached to the pod.
- `volumeMounts` wire/mount the pod's available volume(s) to the container, they exist on a container level.
  That's why we have a `mountVolumes` list pointing to the pod's `scratch-volume`.

Volumes can either be _statically created_ beforehand like here in this case,
or they can be _dynamically provisioned_.

Kubernetes supports many types of volumes. A Pod can use any number of volume types simultaneously. Volume types that are supported:
- **Ephemeral volume** types have a lifetime linked to a specific Pod
  When a Pod ceases to exist, Kubernetes destroys ephemeral volumes.
- **Persistent volumes** types exist beyond the lifetime of any individual Pod.
  When a Pod ceases to exist, Kubernetes does not destroy persistent volumes.

For any kind of volume in a given Pod, data is preserved across container restarts.

## Ephemeral Volumes

Some apps need storage volumes but don't care about their persistence across restarts. An example of this is _caching services_ that are sometimes limited by memory size and can move infrequesntly used data into storage outside of memory with little impact on overall performance.

Types of ephemeral volumes:
- `emptyDir`
- `configMap`
- `downwardAPI`
- `secret`
- `image`

Back to the example above; here we are using `emptyDir` which is a specific type of ephemeral volume that is configured in the kubernetes API by default and is created when the pod is assigned to a node.
`emptyDir`s are ephemeral storage that are initially empty as name suggests.
Data inside `emptyDir`s are safe across container crashed/restarts, it is only when a pod is removed from a node for any reason that the data inside them are deleted.
`emptyDir`s are used for:
- scratch space: like for a disk-based merge sort
- checkpoints: for a long computation for crash recovery

## Persistent Volumes (PV, PVC)

Managing storage is a distinct problem from managing compute instances.
Persistent Volumes subsystem provide an API for users and administrators to abstract the details of two things:
1. how storage is provided
2. how storage is consumed

To achieve this separation, 2 new API resources exist:
- `Persistent Volumes` (PV):
	- It is storage in the cluster that is provisioned by admins (statically provisioned by the admin or dynamically provisioned using `storageClasse`s) to users.
	- It's a cluster resource like any other cluster resource.
	- It captures the implementation details of the storage (NFS, iSCSI, or cloud-specific)
- `Persistent Volume Claims` (PVC):
	- It is a request for storage by users.
	- It is similar to pods:
		- Pods consume node resources & can request specific levels of resources (such as CPU and/or Memory).
		- PVCs consume PV resources & can request specific size and access modes (ReadWriteOnce, ReadOnlyMany, ReadWriteMany, ReadWriteOncePod).

---
>**Static vs Dynamic PV provisioning**
>
>A PV can be provisioned statically like so:
> - an administrator makes a big storage medium on the cluster available as PV for teams/users to use.
> - each team/user "claims" their own part (with certain size and/or access mode) which then becomes like their territory
>
>A PV can be provisioned dynamically like so:
> - no dedicated storage is provisioned beforehand
> - a team/user creates a claim to a piece of storage (that does not exist yet)
> - the kubernetes api is responsible for provisioning that volume dedicated only to that claim and that claim only
>
>Dynamic provisioning of PVs is the most used way, because:
> - Static provisioning requires ops work per-volume which does not scale.
> - Proper lifecycle management, when a PVC is deleted, the storage provisioner can automatically cleanup the underlying storage that was provioned to it (controlled by `reclaimPolicy` on the `storageClass` - `delete` vs `retain`).

Let's create our first `PersistentVolumeClaim` and use it in our `mealie` deployment:
```yaml:storage.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mealie-pv-claim                       # <-- creating the PVC
  namespace: mealie
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 500Mi
```

and now update the `mealie` app deployment to use that PVC:
```yaml:deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  labels:
    app: mealie
  name: mealie
  namespace: mealie
spec:
  replicas: 1
  selector:
    matchLabels:
      app: mealie
  template:
    metadata:
      labels:
        app: mealie
    spec:
      containers:
      - image: ghcr.io/mealie-recipes/mealie:v3.17.0

          - containerPort: 9000
        volumeMounts:
          - mountPath: "/app/data"
            name: mealie-pv-storage            # <-- using the volume by name
      volumes:
        - name: mealie-pv-storage              # <-- volume name
          persistentVolumeClaim:
            claimName: mealie-pv-claim         # <-- using the PVC
```

## Storage Classes (SC)

A storage class is also a cluster resource. It is basically a configuration object that references a storage provider and indicates a level of storage quality for users.

A production system may have multiple SCs (split into tiers of service) like so:
- `fast-ssd` → premium NVMe, `reclaimPolicy: Retain`, for databases
- `standard` → cheaper HDD-backed, `reclaimPolicy: Delete`, for scratch/cache
- `replicated` → Longhorn with 3 replicas, for things that must survive node loss

A developer writing a PVC picks the tier by name `storageClassName: fast-ssd` without knowing the underlying disks, replication, cloud APIs, etc. And that abstraction is the whole point.

In k3s there's a pre-existing default storage class called `local-path`:
```bash
❯ k get sc
NAME                   PROVISIONER             RECLAIMPOLICY   VOLUMEBINDINGMODE      ALLOWVOLUMEEXPANSION   AGE
local-path (default)   rancher.io/local-path   Delete          WaitForFirstConsumer   false                  30d
```

This is not the case for a bare `kubeadm` k8s cluster, k8s ships with no default SC. If a developer creates a PVC on a bare `kubeadm` cluster, it will stay `Pending` forever since nothing is there to provision the storage.

The main way we are going to use SCs is that the platform we are deploying to (cloud, on-prem) is going to have storage providers and we'll create different SCs for different usecases and then PVCs are going to use them.
